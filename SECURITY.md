# Security Policy

Tray processes your clips, files and notes **entirely on-device**. It does not upload your data, and the shipped build contains no telemetry. Tray never synthesizes paste keystrokes into other apps — clicking a clip only writes it back to the system pasteboard.

## How your data is protected

**Encryption at rest.** Clipboard history is persisted to the app's sandbox container at `…/Application Support/Tray/history.enc` as a single **AES-GCM (256-bit, authenticated)** blob via Apple's CryptoKit. The file is ciphertext + authentication tag — it cannot be read or tampered with without the key, and a stray copy of the file is not a plaintext dump.

**Key management.** The 256-bit data-encryption key is generated with CryptoKit on first use and stored in the **macOS Keychain** as a generic-password item, with `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` (device-bound, never synced to iCloud). It is never persisted in a file alongside the encrypted data.

- The first time Tray reads or writes that key, macOS may present a Keychain access prompt. This is expected and is macOS gating access to the key; choosing **“Always Allow”** lets a stable, signed build access it without re-prompting. Local development builds are re-signed on each rebuild, so they may re-prompt.

**Passcode (private mode).** An optional passcode masks clip contents in the UI. The passcode itself is **never stored**; only `PBKDF2-HMAC-SHA256` with a **16-byte random salt and 120,000 iterations** is kept (in the Keychain). Verification is a constant-time comparison. A forgotten passcode cannot be recovered — it can only be removed (which discards the masking, not the history).

**Sensitive-clip filtering.** Items that other apps flag as concealed or transient are never recorded. Tray honors the `nspasteboard.org` convention (`org.nspasteboard.ConcealedType`, `TransientType`, `AutoGeneratedType`) plus a default ignore list (1Password, KeeWeb, TextExpander/Butler, Typinator, Apple Universal Clipboard).

**Sandbox.** Tray runs inside the macOS App Sandbox with only user-selected file access and outgoing network (for the anonymous version check). It cannot read other apps' windows or inject input.

## Threat model (what the protections do and don't cover)

- **Covered:** a stolen/copied `history.enc` (it's encrypted); casual or over-the-shoulder viewing while Tray is open (private-mode masking); recording of password-manager clips (filtered out); data exfiltration (there is none — fully local).
- **Not covered:** an attacker with control of your **unlocked** macOS user session can read the Keychain key and therefore decrypt history, exactly like any local clipboard manager. The passcode is a UI privacy gate, **not** a second encryption factor — it does not derive the data key. Hardening this (wrapping the data key with a passcode-derived key) is tracked as a possible future enhancement.

## Reporting a vulnerability

Please report security issues **privately** — open a [GitHub security advisory](https://github.com/guptaprakhariitr/tray-mac/security/advisories/new) or email the address on the maintainer's GitHub profile — rather than filing a public issue. Please include reproduction steps and the affected version. We aim to acknowledge within a few days and will credit reporters who wish to be named.

## Notes

- `GoogleService-Info.plist` and any signing keys are never committed (see `.gitignore`).
- Encryption uses only standard, platform-provided algorithms (AES-GCM via CryptoKit, PBKDF2 via CommonCrypto) to protect the user's own on-device data — see the export-compliance note in `docs/APP_STORE_SUBMISSION.md`.
- Tray runs inside the macOS App Sandbox with only user-selected file access; it does not read other apps' windows or inject input.
